Communications Hub implementation spec

Stack: Flask + SQLAlchemy + Jinja + existing app patterns
Goal: evolve the current SMS-only module into a unified customer communications hub that supports:

customer-level communication screen with template selection

5 actions:

MicroSMS send

SMS via phone

Call via phone

WhatsApp

Viber

bulk MicroSMS sending from Order Review

logging of every communication action

1. Core design decision

Do not replace the current MicroSMS implementation.
Keep it and build a generic communications layer above it.

Keep existing pieces

Reuse these from the current sms module:

_normalize_mob

_is_valid_mob

_render_db_template

_microsms_send_direct

_parse_provider_response

DLR callback logic

existing sms_template

existing sms_log for backward compatibility

Add new generic layer

Create a new communications module and new logging table(s) so we can support:

provider sends

phone launches

calls

WhatsApp

Viber

manual outcome updates

From the order review queue:

allow selecting many customers with checkboxes

add bulk action: Send MicroSMS

create one batch header record

C. Logging

Every contact must be logged.

That includes:

MicroSMS send attempt

Phone SMS launch

Call launch

WhatsApp launch

Viber launch

For manual channels, launching the action must still create a log record, even if delivery cannot be guaranteed.

Also support manual outcome updates, especially for calls:

answered

wrong number

left message

discussed order

3. Database changes
3.1 New table: crm_communication_log

Create a generic communication log.

CREATE TABLE IF NOT EXISTS crm_communication_log (
    id BIGSERIAL PRIMARY KEY,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),

    created_by_username VARCHAR(100) NULL,

    customer_code_365 VARCHAR(50) NULL,
    customer_name VARCHAR(255) NULL,

    context_type VARCHAR(50) NULL,      -- customer, order_review, crm_task, collections, etc
    context_id VARCHAR(100) NULL,

    source_screen VARCHAR(50) NULL,     -- customer_profile, order_review, crm_dashboard

    channel VARCHAR(30) NOT NULL,       -- microsms, phone_sms, phone_call, whatsapp, viber
    direction VARCHAR(10) NOT NULL DEFAULT 'outbound',

    template_code VARCHAR(100) NULL,
    template_title VARCHAR(255) NULL,

    recipient_number VARCHAR(50) NULL,
    recipient_number_normalized VARCHAR(30) NULL,

    message_text TEXT NULL,

    status VARCHAR(30) NOT NULL DEFAULT 'initiated',
    outcome_note TEXT NULL,

    batch_id VARCHAR(120) NULL,
    provider_name VARCHAR(50) NULL,
    provider_message_id VARCHAR(120) NULL,
    provider_error_code VARCHAR(50) NULL,
    provider_raw_response TEXT NULL,

    dlr_status VARCHAR(50) NULL,
    dlr_received_at TIMESTAMP NULL,

    launch_url TEXT NULL,               -- whatsapp/viber/tel/sms link used
    extra_json JSONB NULL
);
Indexes
CREATE INDEX IF NOT EXISTS idx_comm_log_customer
    ON crm_communication_log (customer_code_365, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_comm_log_batch
    ON crm_communication_log (batch_id);

CREATE INDEX IF NOT EXISTS idx_comm_log_context
    ON crm_communication_log (context_type, context_id, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_comm_log_channel_status
    ON crm_communication_log (channel, status, created_at DESC);
3.2 New table: crm_communication_batch

Needed for bulk actions.

CREATE TABLE IF NOT EXISTS crm_communication_batch (
    id BIGSERIAL PRIMARY KEY,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    created_by_username VARCHAR(100) NULL,

    source_screen VARCHAR(50) NULL,      -- order_review
    channel VARCHAR(30) NOT NULL,        -- microsms
    template_code VARCHAR(100) NULL,
    template_title VARCHAR(255) NULL,

    total_selected INTEGER NOT NULL DEFAULT 0,
    total_valid INTEGER NOT NULL DEFAULT 0,
    total_sent INTEGER NOT NULL DEFAULT 0,
    total_failed INTEGER NOT NULL DEFAULT 0,
    total_skipped INTEGER NOT NULL DEFAULT 0,

    batch_id VARCHAR(120) NOT NULL UNIQUE,
    notes TEXT NULL,
    extra_json JSONB NULL
);

6.2 normalize_phone(number, default_country='CY') -> dict

Return a structured dict:

{
    "raw": "...",
    "normalized": "99123456",
    "digits_only": "99123456",
    "display": "+357 99 123456",
    "e164": "+35799123456",
    "valid": True,
    "reason": None,
}
Cyprus rule

If the number is 8 digits and looks like a Cyprus local mobile/phone, derive +357.

Use this helper for:

MicroSMS

sms links

WhatsApp links

18. Important implementation notes
Phone formatting

This is critical for WhatsApp/Viber/tel/sms links.
Do not rely only on current digits-only validation.
Implement one normalization helper and use it everywhere.

Desktop vs mobile

On desktop:

phone/call/WhatsApp/Viber launch may not always open the target app

still log the action

provide copy number / copy message fallback

Template preview

The message shown on screen should be editable before sending for customer-level actions.
For bulk send:

keep template-based rendering per customer

do not allow per-customer manual edits in the first version

Bulk safety

Before sending bulk MicroSMS:

show count of valid and invalid recipients

skip invalid ones

do not abort the whole batch בגלל one bad customer

20. Final implementation principle

The new system should behave like this:

customer-level

open customer → choose template → choose channel → action is launched/sent → action is logged → history visible immediately

operational bulk

select customers from order review → choose MicroSMS template → send to all valid numbers → each customer logged individually → one batch summary exists
