Build objective

Add a full “Offer Intelligence” layer to EP WMDS CRM so that customer-specific Magento offers can be reviewed:
1) on the main CRM dashboard
2) on Review Ordering
3) inside a customer offer drawer/panel with KPIs and detailed offer lines

Use Flask + SQLAlchemy.
Do not break or redesign the current order-window logic.
Keep the existing routes and behavior, and extend them cleanly.

--------------------------------------------------
1. BUSINESS GOAL
--------------------------------------------------

We already import daily Magento customer offer rows with these columns:

- snapshot_at
- customer_id
- customer_email
- sku
- product_name
- rule_code
- rule_name
- rule_description
- origin_price
- customer_final_price

We want to turn this into CRM intelligence, not just a raw price list.

The CRM user must be able to see quickly:
- whether the customer has active offers
- how many active offer SKUs they have
- average / strongest discount
- whether they used these offers recently
- which offered SKUs are not being bought
- whether any active offers are low-margin / risky
- which offer lines are the best opportunity to mention in a call / visit / push / SMS

We also need cost enrichment from our item master so we can calculate:
- gross profit
- gross margin %
- margin risk flags

IMPORTANT:
Do not load detailed offer rows into the main dashboard query.
The main dashboard and review-ordering pages must only join a compact customer summary table.
Detailed offer lines must load lazily on click.

--------------------------------------------------
2. IMPLEMENTATION PRINCIPLES
--------------------------------------------------

- Keep the raw Magento import exactly as received for audit/debugging.
- Normalize current offers into a proper current table.
- Build a current customer summary table for fast CRM queries.
- Use lazy-loaded detail endpoints for per-customer offer lines.
- Reuse the current dark WMDS style.
- Do not add heavy joins against raw import data in dashboard routes.
- Keep pagination and list performance good.

--------------------------------------------------
7. KPI LOGIC
--------------------------------------------------

Use these calculations.

7.1 Line-level calculations
For each offer line:
- discount_value = origin_price - offer_price
- discount_percent = ((origin_price - offer_price) / origin_price) * 100 if origin_price > 0 else null
- gross_profit = offer_price - cost if cost is not null else null
- gross_margin_percent = ((offer_price - cost) / offer_price) * 100 if offer_price > 0 and cost is not null else null

7.2 Configurable thresholds
Add Settings keys:
- crm_offer_low_margin_pct_threshold default 12
- crm_offer_negative_margin_pct_threshold default 0
- crm_offer_strong_discount_pct_threshold default 15
- crm_offer_unused_days_threshold default 30
- crm_offer_recent_sales_days default 28

7.3 margin_status
- unknown -> cost or offer price missing
- negative -> gross_margin_percent < 0
- low -> gross_margin_percent >= 0 and < low_margin_threshold
- healthy -> >= low_margin_threshold

7.4 line_status
- selling -> sold_qty_4w > 0
- high_discount_unused -> discount_percent >= strong_discount_threshold and sold_qty_4w = 0
- margin_risk -> margin_status in ('low','negative')
- unused -> sold_qty_4w = 0
- unknown -> fallback

7.5 Summary KPIs per customer
- has_special_pricing = active_offer_skus > 0
- active_offer_skus = count distinct sku
- active_offer_rules = count distinct rule_code
- avg_discount_percent = avg(discount_percent)
- max_discount_percent = max(discount_percent)
- avg_gross_margin_percent = avg(gross_margin_percent where not null)
- margin_risk_skus = count where margin_status in ('low','negative')
- negative_margin_skus = count where margin_status = 'negative'
- offered_skus_bought_4w = count distinct sku where sold_qty_4w > 0
- offered_skus_bought_90d = count distinct sku where sold_qty_90d > 0
- offered_skus_not_bought = active_offer_skus - offered_skus_bought_4w
- offer_sales_4w = sum(sold_value_4w)
- offer_sales_90d = sum(sold_value_90d)
- offer_utilisation_pct = offered_skus_bought_4w / active_offer_skus * 100
- high_discount_unused_skus = count where line_status = 'high_discount_unused'
- top_rule_name = most frequent rule_name by offer count
- top_opportunity_count = count of lines where line_status in ('unused','high_discount_unused') and margin_status != 'negative'
