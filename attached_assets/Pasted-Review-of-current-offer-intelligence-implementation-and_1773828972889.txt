Review of current offer-intelligence implementation and exact next steps

Overall status

The backend is moving in the right direction, but it is not finished yet.
The main issues now are:
1) duplicated old/new offer schemas
2) missing resolver logic fallback
3) incomplete KPI aggregation
4) no refresh locking
5) no front-end offer drawer/UI yet
6) a few correctness bugs in the service layer

Do the work below in this order.

==================================================

==================================================

Keep these tables as the official offer-intelligence source of truth:
- crm_customer_offer_import_batch

- crm_customer_offer_summary_current
- crm_customer_offer_unresolved

Do NOT continue building new logic around these old legacy tables:

Before any import/rebuild, the new tables/indexes/constraints must exist.

3) Add refresh locking to price offers refresh

The route /crm/price-offers/refresh currently has no lock.
That means two users or a cron+user can run the rebuild at the same time.

Implement the same lock pattern used in open orders refresh.

Add a job name like:
- JOB_NAME = "crm_price_offers_refresh"

Create helper functions similar to open orders sync lock if a generic lock service already exists:
- acquire_sync_lock()
- release_sync_lock()

Expected behavior:
- if refresh already running, return HTTP 409
- otherwise run refresh and release lock in finally block

4) Fix JSON body handling in refresh route

Current code can fail if request.json is None.

Change:
- payload = request.get_json(silent=True) or {}
- csv_path = payload.get("csv_path")

Do not use request.json.get(...) directly.

8) Make rule_code non-null and normalized

Current issue:
Unique constraint is:
(customer_code_365, sku, rule_code)

If rule_code is NULL, Postgres unique behavior can allow duplicates.
That can create duplicate rows for the same customer+sku where rule_code is null.

Fix:
Normalize rule code during import/rebuild:
- normalized_rule_code = stripped rule_code if present
- otherwise use '__NO_RULE__'

Also normalize rule_name similarly if blank:
- rule_name = original if present else 'No Rule'

Then make rule_code consistently non-null inside crm_customer_offer_current.

Expected result:
- one stable row per customer + sku + normalized rule

9) Remove redundant DELETE + ON CONFLICT pattern

Current issue:
_rebuild_from_batch() does:
- DELETE FROM crm_customer_offer_current
- then INSERT ... ON CONFLICT ...

If current table is fully rebuilt each run, ON CONFLICT is unnecessary.

Simpler and faster:
- DELETE FROM crm_customer_offer_current
- bulk INSERT fresh rows only

If you keep ON CONFLICT, then do not clear the whole table first.
Choose one strategy only.

Recommendation:
Because current table represents latest snapshot only:
- clear current table
- bulk insert fresh rows
- no ON CONFLICT

This will be cleaner and faster.
