Replit implementation brief — Forecasting fixes phase 1

Work only inside the live forecasting code under services/forecast/.

Do not redesign the engine.
Do not change the weekly sales builder structure.
Do not change the overall run order.

The goal of this phase is to make the existing forecasting engine more consistent, more deterministic, and safer operationally.

Scope of this change

Implement only these 3 fixes:

Align new_sparse forecast method naming with actual behavior
Use one supplier source of truth in replenishment
Rebuild review flags/reasons from scratch on every run

Do not expand beyond that in this phase.

1. Fix new_sparse method inconsistency
Problem

classification_service.py labels new_sparse items with a forecast method like AVG_NZ, but base_forecast_service.py actually uses a seeded analogue forecast.

This causes confusion in:

stored profile data
explainability
debugging
UI trust
Required change

Make the declared method match the actual calculation.

Decision

Keep the current seeded analogue logic in base_forecast_service.py.

Change the method label from AVG_NZ to:

SEEDED

or if you prefer more explicit naming:

SEEDED_ANALOGUE

I recommend SEEDED because it is shorter and cleaner for UI.

Files to update
services/forecast/classification_service.py
any forecast result / profile persistence code that stores forecast_method
any enums/constants/helpers if they exist
any UI serializer or API payload that exposes method labels
Required logic

Where demand class is new_sparse:

forecast method must be stored as SEEDED
never store AVG_NZ for that class unless the engine truly switches to that formula
Acceptance criteria

For any new_sparse item:

classification output shows forecast_method = "SEEDED"
forecast result also shows forecast_method = "SEEDED"
there is no mismatch between profile and actual forecast engine
2. Unify supplier source of truth in replenishment
Problem

Current replenishment appears to use:

one source for stock fetching, likely DwItem.supplier_code_365
another source for planning parameters, likely ForecastItemSupplierMap

This can create wrong results if the two do not match.

Required change

Create one supplier-resolution step per item and use it consistently for:

stock fetch
lead time
review cycle
MOQ
order multiple
any supplier-based planning values
Rule

Resolve supplier in this priority order:

Primary

ForecastItemSupplierMap active/default mapping for the item

Fallback

DwItem.supplier_code_365

If neither exists:

mark item for review
skip replenishment calculation or continue with safe defaults, depending on current system behavior
log clearly
Implementation detail

Add a helper in replenishment_service.py, for example:

_resolve_supplier_context(item_code_365)

It should return a structured object/dict with:

supplier_code
source (supplier_map or dw_item)
lead_time_days
review_cycle_days
min_order_qty
order_multiple
any other supplier planning fields already used

Then all replenishment logic must use this resolved context only.

Do not separately re-read supplier from another place later in the flow.

Logging

For each item where supplier resolution falls back or fails, log clearly:

item code
chosen supplier code
source used
whether fallback was used
Acceptance criteria

For a given item:

the same supplier code is used for stock fetch and planning parameters
no mixed-source supplier behavior remains
fallback behavior is deterministic and logged
3. Rebuild review flags and reasons from scratch every run
Problem

Review flags/reasons may accumulate stale text across runs.

Examples:

item was erratic last run, smooth this run
old warning remains attached
review status becomes noisy and unreliable
Required change

At the start of forecast evaluation for each item, reset review fields and rebuild them deterministically from current run state only.

Rule

Never append onto old persisted review text.

Instead:

start with empty list of review reasons
evaluate current conditions
build a fresh list
set:
review_flag = bool(reasons)
review_reason = "; ".join(reasons) or equivalent structured format already used
Suggested review triggers

Keep your existing business triggers, but rebuild them cleanly. For example:

new/sparse demand
erratic demand
lumpy demand
no demand but active item
missing supplier mapping
missing stock data
missing lead time
fallback supplier used
forecast confidence low
any current replenishment exception already in logic

Do not duplicate the same reason twice.

Implementation detail

In the forecast/replenishment flow, do something like:

create review_reasons = []
add current-run reasons only
normalize/deduplicate
save final values

If review building happens in more than one service, centralize it in one helper to avoid drift.

Example helper:

_build_review_state(...)
Acceptance criteria

When an item changes condition between runs:

old review text disappears
only current valid reasons remain
review_flag matches the rebuilt reasons exactly
