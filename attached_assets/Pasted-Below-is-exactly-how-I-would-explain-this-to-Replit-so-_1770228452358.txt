1) Clarify the 3 “flags/statuses” you see
A) Discrepancy Open

Meaning: There is at least one discrepancy recorded for the invoice that is not resolved.

Rule (query logic):

EXISTS delivery_discrepancies WHERE invoice_no = X AND is_resolved = false

B) Post-Delivery Case Open

Meaning: The invoice requires back-office handling (credit note, investigation, returns check, etc.).

Rule:

EXISTS invoice_post_delivery_cases WHERE invoice_no = X AND status IN ('OPEN','IN_PROGRESS','PENDING_CN') (whatever you standardize)

Creation trigger: when the first discrepancy is created for an invoice.

C) Credit Note Required

Meaning: at least one discrepancy implies a credit note must be issued (missing/damaged/wrong/short pick etc.), but it’s not issued yet.

Rule (recommended):

Store this at the invoice case level:

invoice_post_delivery_cases.credit_note_required = true

invoice_post_delivery_cases.credit_note_no IS NULL (or empty)

Creation trigger: when a discrepancy type that requires CN is recorded.

Important implementation note (to keep schema simple):

For damaged, set:

actual_item_code = item_code_expected

actual_qty = qty_actual (or qty returned)

For extra_item / over_pick, set:

qty_expected = 0

actual_item_code = same as expected

actual_qty = qty returned

This lets you use one structure for line-level returns without adding a new table.

3) Critical missing piece: store the money impact per discrepancy

Right now your delivery_discrepancies table has no amount fields, but your process requires:

“Driver deducts that amount from the collection based on the value on the invoice.”

So Replit must add to delivery_discrepancies:

deduct_amount numeric(12,2) NOT NULL DEFAULT 0

currency varchar(3) DEFAULT 'EUR' (optional)

How it’s filled:

When driver records discrepancy, system auto-looks-up the line value (from your invoice line source: PS365 / DW invoice lines / existing invoice data), calculates the amount for the affected qty, and stores it.

Driver can be allowed to adjust within limits (admin policy), but the stored value becomes the audit source.

Without this, reconciliation will always be “manual math”.

6) “Unsettled invoice” definition (for POD invoices)

You want:

Expected group = POD vs CREDIT

POD breakdown is by actual payment method

online + post-dated remain pending

discrepancies reduce what is due now

Compute per invoice (DELIVERED only)

A) Determine expected group

If invoice is credit terms → group = CREDIT

Else → group = POD

B) Compute due_now

due_now = invoices.total_grand - SUM(discrepancies.deduct_amount where discrepancy implies deduction)

C) Allocate payments
This is the other big requirement:

Your current cod_receipts has invoice_nos JSON, which makes per-invoice allocation hard if one receipt covers multiple invoices.

Minimum rule to implement immediately:

In driver UI, enforce one COD receipt record per invoice (even if the customer pays 3 invoices together).

Same cheque number can be reused, but each invoice gets its own row.

Then each invoice has clear received_amount and method.

Now you can classify:

Cash collected = sum COD rows where method=CASH

Day cheque collected = method=DAY_CHEQUE

Pending online = method=ONLINE (pending)

Pending post-dated = method=POST_DATED (pending)

D) Unsettled invoice
An invoice is “unsettled” if:

group = POD

status = delivered

and due_now != (cash + day_cheque + pending_online + pending_postdated)
OR any pending exists (depending on how strict you want it)
