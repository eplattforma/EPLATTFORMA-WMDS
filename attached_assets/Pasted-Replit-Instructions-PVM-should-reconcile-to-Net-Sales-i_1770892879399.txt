1) Backend: extend /pricing/api/pvm summary to include CN/credits totals
File

blueprints/pricing_analytics.py

Change

Inside api_pvm() keep your existing PVM SQL as-is (it explains Gross Positive Sales).

Add two extra totals queries: one for current period and one for baseline period, that compute:

net_sales (includes CN/returns): sum(net_excl) where qty <> 0

gross_positive_sales: sum(net_excl) where qty > 0 AND net_excl > 0

credits_sales: sum(net_excl) where net_excl < 0 OR qty < 0 (this will be negative)

1.1 Add this helper SQL (inside api_pvm)

Paste near the top of api_pvm() after you validate compare mode:

totals_sql = text("""
  SELECT
    COALESCE(SUM(CASE WHEN qty <> 0 THEN net_excl ELSE 0 END), 0) AS net_sales,
    COALESCE(SUM(CASE WHEN qty > 0 AND net_excl > 0 THEN net_excl ELSE 0 END), 0) AS gross_positive_sales,
    COALESCE(SUM(CASE WHEN (net_excl < 0 OR qty < 0) THEN net_excl ELSE 0 END), 0) AS credits_sales
  FROM dw_sales_lines_mv
  WHERE sale_date BETWEEN :d_from AND :d_to
    AND customer_code_365 = :customer
""")

1.2 Execute it for current + baseline

After you run your existing PVM query and build rows, add:

cur_tot = db.session.execute(totals_sql, {
    "customer": customer, "d_from": d_from, "d_to": d_to
}).mappings().first()

base_tot = db.session.execute(totals_sql, {
    "customer": customer, "d_from": b_from, "d_to": b_to
}).mappings().first()

net_cur = float(cur_tot["net_sales"] or 0)
net_base = float(base_tot["net_sales"] or 0)

gross_cur = float(cur_tot["gross_positive_sales"] or 0)
gross_base = float(base_tot["gross_positive_sales"] or 0)

credits_cur = float(cur_tot["credits_sales"] or 0)   # usually negative
credits_base = float(base_tot["credits_sales"] or 0)

Those sums can be impacted by your LIMIT and by missing tail items.

Replace your “revenue_current / revenue_baseline” in the summary with the totals query values:

Use gross_cur/gross_base as the “PVM explained revenue”

4) Why this approach is the right one

If you include CN directly inside PVM unit price math:

you create negative quantities and negative revenues,

unit prices become nonsensical,

“price effect” becomes misleading (because credits aren’t a price change).

With the split approach you get:

correct accounting reconciliation (net totals match 360)

correct commercial explanation (P/V/M on real sales)
