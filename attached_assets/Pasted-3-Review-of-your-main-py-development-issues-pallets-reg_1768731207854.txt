3) Review of your main.py (development issues + pallets registration)

Your main.py is structurally fine for your pattern (blueprints + runtime schema updates). But I would change these items immediately because they will bite you in development and production:

A) PERMANENT_SESSION_LIFETIME should be a timedelta, not an int

Flask expects a timedelta. Use:

from datetime import timedelta

app.config.update({
    "PERMANENT_SESSION_LIFETIME": timedelta(hours=1),
})

B) DEBUG should follow is_production

Right now you hard-set DEBUG: False and then run with debug=True at the bottom. Make it consistent:

app.config.update({
    "DEBUG": (not is_production),
})

C) Your local_time_filter() should handle naive datetimes safely

If any timestamp is naive, dt.astimezone() will crash. Protect it:

from datetime import timezone

if dt.tzinfo is None:
    dt = dt.replace(tzinfo=timezone.utc)

D) Pallets blueprint prefix

You registered:

app.register_blueprint(pallets_bp, url_prefix='/routes')


That is OK only if all pallets endpoints are something like:

/routes/<shipment_id>/pallets

/routes/<shipment_id>/pallets/create

/routes/pallets/<pallet_id>/assign

Avoid defining any pallets route as @bp.route('/<int:shipment_id>') because that would collide with your existing routes blueprint.

4) Proposed palletisation model (matches your photos and your 1/8 block rule)
Key rule (phase 1)

A Euro pallet base is split into 8 fixed blocks.

Each block = 300 × 400 mm

Each order occupies 1..8 blocks, preferably contiguous (horizontal separation).

Constraints:

height ≤ 1.8m (you said 1.7/1.8; store 1.8 as max, use 1.7 as operational “warning” threshold if you want)

weight ≤ 500kg

“Cases corridors 09–12” drive the base layer weight and “bottom-first” advice.

What the picker needs (phase 1 output per order)

For each order (invoice):

Suggested blocks (1..8)

Handling warnings:

fragile yes/semi/medium

spill risk

pressure high (do not crush)

heat sensitive (cooler bag)

“Bottom material” summary:

base weight coming from corridor 09–12 items (or OI BOTTOM)

5) How to estimate order blocks (simple + robust with your available columns)

You actually have item_length/width/height in ps_items_dw (many are 0 but still useful). Use this hybrid:

For each line:

line_weight_kg = InvoiceItem.line_weight else qty * DwItem.item_weight

unit_volume_m3:

if L/W/H > 0: (L*W*H) * 1e-6 (assuming cm)

else fallback: line_weight_kg / 300 (density proxy 300 kg/m³)

For the invoice:

total_volume_m3 = sum(qty * unit_volume_m3)

required_footprint_m2 = total_volume_m3 / effective_stack_height

use effective_stack_height = 1.5 (realistic packing height)

pallet_area_m2 = 0.8 * 1.2 = 0.96

fraction = required_footprint_m2 / pallet_area_m2

suggested_blocks = ceil(fraction * 8 * inefficiency_factor)

inefficiency_factor = 1.15 (cartons/voids)

Weight floor:

suggested_blocks = max(suggested_blocks, ceil(total_weight_kg / 62.5)) (500/8)

Clamp to 1..8.

This will give you a defensible “first stage” estimation that you can calibrate later.
