# Final Cooler Box Workflow Implementation Brief

## Main Objective

Finalise the cooler box process as a **warehouse cooler packing and handover workflow**.

The system must support this process:

1. Warehouse opens **Cooler Packing / Handover** from the Warehouse menu.
2. Warehouse selects route/date.
3. Warehouse confirms the delivery stop sequence.
4. Warehouse generates a cooler box plan.
5. The planner uses the selected **Max Fill %** and available box counts.
6. The planner uses the **fewest possible boxes** without exceeding selected Max Fill %.
7. Box 1 contains the **last delivery stops**.
8. Picker starts Box 1, picks everything assigned to Box 1, then closes/seals it.
9. Picker continues Box 2, Box 3, etc.
10. After all boxes are picked and closed, the route becomes **Cooler Ready for Dispatch**.
11. Warehouse can access completed cooler boxes from Warehouse menu, not only from Delivery Route.

# 12. Cooler box assignment algorithm

## Existing parameter

The UI already has a fill parameter.

It may currently be called:

```text
fillTargetSelect
target_fill
target_fill_pct
```

Keep the parameter but clarify the meaning.

Rename where practical to:

```text
max_fill_pct
```

## Business meaning

The selected Max Fill % means:

```text
Use as few boxes as possible from the available boxes, without filling any box above the selected percentage.
```

Example:

```text
User selects 80% → max_fill_pct = 0.80
User selects 90% → max_fill_pct = 0.90
User selects 75% → max_fill_pct = 0.75
```

This is parametric.

Do not hard-code `0.80` except as a fallback default.

## UI requirement

In Box Planning, show:

```text
Max Fill %
```

Options:

```text
70%
75%
80%
85%
90%
95%
100%
```

Default:

```text
80%
```

Help text:

```text
Use as few boxes as possible, without filling any box above the selected percentage.
```

## Server-side validation

Validate selected Max Fill %:

```text
allowed range: 0.50 to 1.00
default: 0.80 if missing/invalid
```

If selected value is high, for example above 90%, optionally show warning or require manager permission.

# 13. Correct algorithm objective

The planner objective must be:

```text
1. Assign all eligible cooler items.
2. Use the fewest possible boxes.
3. Use only available box types/counts.
4. Do not exceed selected Max Fill % for volume or weight.
5. Keep Box 1 as the last delivery stops.
6. Keep delivery stops together where possible.
7. Avoid splitting a stop unless the stop cannot fit in one available box under the selected Max Fill %.
8. Avoid splitting a queue line unless quantity splitting is explicitly implemented.
9. Among plans with the same number of boxes, prefer:
   - fewer split stops
   - better fill utilisation
   - smaller suitable boxes where appropriate
```

The planner must not create extra boxes just to achieve a nicer fill percentage.

Fewer boxes is the primary objective, as long as max fill and availability constraints are respected.

---

# 14. Capacity calculation

Treat selected Max Fill % as a hard capacity constraint.

For each box type:

```text
usable_volume = box.volume * max_fill_pct
usable_weight = box.max_weight * max_fill_pct
```

A box is valid only if:

```text
planned_volume <= usable_volume
planned_weight <= usable_weight
```

Do not silently exceed selected Max Fill %.

Do not silently exceed available box counts.

# 15. Availability enforcement

If user enters available box counts, the planner must not create more boxes than available.

Example:

```text
Small boxes available: 2
Plan requires Small boxes: 3
```

Return error:

```text
Plan requires 3 Small boxes but only 2 are available.
```

If no valid plan exists, return:

```text
No valid box plan found using the selected Max Fill % and available boxes.
```

Include:

```text
estimated total volume
estimated total weight
available capacity under selected Max Fill %
shortage details
suggested actions
```

Suggested actions:

```text
increase available boxes
use larger boxes
increase Max Fill % with manager approval
split oversized stops/items manually
```

Only allow over-allocation or max-fill override if explicitly implemented as manager override.

# 16. Fewest-box planning approach

The planner should not remain purely greedy if that prevents the fewest valid boxes.

Preferred approach:

1. Build eligible route items sorted by delivery sequence last-first.
2. Calculate stop/item volume and weight.
3. Build available box candidates from selected box counts.
4. Try to find a valid plan using 1 box.
5. If not possible, try 2 boxes.
6. Continue until all available boxes are exhausted.
7. Select the first valid plan with the fewest boxes.
8. If multiple plans use the same box count, score them.

Scoring for same box count:

```text
fewer split stops = better
higher average fill = better
less unused capacity = better
smaller suitable boxes preferred over oversized boxes
```

Important:

```text
Box 1 must always cover the last delivery stops.
Box 2 covers the next previous delivery stops.
Box 3 covers the next previous delivery stops.
```
