# EP WMDS — Account Manager Cockpit
## Implementation Brief for Replit

**Status:** Ready for implementation
**Approach:** Additive only. Build a new comprehensive customer view as a new menu item. Leave all existing customer reports untouched. Consolidation decisions are deferred until after this is in real use for several weeks.

---

## 1 — Goal

Build a single new page — the **Account Manager Cockpit** — that gives sales managers, account managers, and sales reps a comprehensive, action-oriented view of one customer's commercial performance. The page combines existing data (KPIs, top items, peer comparisons, offers, white space, lapsed items, price index, activity timeline) with **new analytics and capabilities that don't exist anywhere in the codebase today**:

- Editable spend targets at four cadences (weekly ambition, monthly, quarterly, annual) with manager-approval workflow
- "Offers he could get" — items the customer regularly buys with no offer assigned, where peers in the same reporting group have one
- GP-based item ranking (codebase ranks by revenue today)
- PVM (Price/Volume/Mix) decomposition vs prior year
- Per-category churn risk
- Composite engagement score
- Greek-language Claude-generated recommendations specifically tied to closing the gap-to-target

The page is **purely additive**. Every existing customer report (`/crm/dashboard`, `/crm/review-ordering`, `/abandoned-carts`, `/customer-benchmark`, `/analytics/customers/<code>`, `/peer-analytics/<code>`, `/pricing-analytics/customer/<code>`, `/analytics/top-opportunities`, `/analytics/`) continues to work exactly as today. No deprecation, no redirects, no menu reshuffle. The cockpit appears as one new menu item alongside the existing ones.

After 4–6 weeks of real use we'll revisit which existing reports the cockpit replaces. That decision is **out of scope for this brief**.

## 4 — Architectural rules

These apply to every ticket below.

1. **One new blueprint at `/cockpit/`.** Single URL prefix for everything new. Registered in `main.py` next to other customer-report blueprints, with defensive try/except like the `cypost_bp` block.
2. **Same canonical data, no copies.** Cockpit reads from existing tables (`dw_invoice_lines`, `dw_customers`, `crm_customer_offer_current`, `dw_items`, etc.). Only new schema objects: SQL views and the targets tables (Section 6).
3. **`@require_permission` on every route.** No hardcoded role lists. No `current_user.role` checks. New permission keys are listed in Section 5.
4. **`display_name` everywhere user-visible.** Never display raw `username`. If a Jinja filter for this doesn't exist from Phase 1, create `user_display` filter.
5. **Two LLM providers, one cache table.** OpenAI service stays untouched and serves the legacy "Ask AI" button. New `claude_advice_service.py` serves the cockpit. They share `ai_feedback_cache` with a key prefix to avoid collisions.
6. **Reuse, don't reimplement.** Existing query helpers in `routes_customer_analytics.py`, `routes_customer_benchmark.py`, `blueprints/peer_analytics.py`, `routes_pricing_analytics.py`, `services/crm_price_offers.py`, `blueprints/abandoned_carts.py` are called as Python functions where they're cleanly factored. Where logic is inlined in route handlers, replicate the SQL faithfully in `services/cockpit_data.py`. Do not modify the originals.
7. **Settings table for feature flag.** `cockpit_enabled` (bool, default false) controls visibility. Documented in `ROLLBACK_AND_FLAGS.md`.
8. **Additive migrations only.** New columns nullable. New tables independent. Every migration must have a documented rollback (drop table / drop column / drop view).

## 6 — Customer Spend Targets (the editable feature)

This is the single most important new capability. AMs and sales managers set spend targets at four cadences. The cockpit shows actual vs target, gap, and run-rate projection. The cockpit's recommended actions are tuned to closing the gap.

### 6.1 — Schema

Two new tables. Both fully reversible (drop table). Nullable additive columns where they touch existing tables.

```sql
-- Active targets per customer.
-- One row per customer. Targets can be NULL initially.
CREATE TABLE customer_spend_target (
    customer_code_365      VARCHAR(50) PRIMARY KEY,
    target_weekly_ambition NUMERIC(12,2),    -- "what we'd love to see in a typical order"
    target_monthly         NUMERIC(12,2),
    target_quarterly       NUMERIC(12,2),
    target_annual          NUMERIC(12,2),
    status                 VARCHAR(20) NOT NULL DEFAULT 'active',
                                             -- 'active' (approved & in effect)
                                             -- 'proposed' (AM-submitted, awaiting manager approval)
    proposed_by            VARCHAR(64),      -- if status='proposed', who submitted it
    proposed_at            TIMESTAMP WITH TIME ZONE,
    proposed_notes         TEXT,
    approved_by            VARCHAR(64),
    approved_at            TIMESTAMP WITH TIME ZONE,
    last_modified_at       TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    last_modified_by       VARCHAR(64) NOT NULL,
    FOREIGN KEY (customer_code_365) REFERENCES dw_customers(customer_code_365)
);

CREATE INDEX idx_customer_spend_target_status ON customer_spend_target(status);

-- Full audit history. Every change creates a row.
CREATE TABLE customer_spend_target_history (
    id                     SERIAL PRIMARY KEY,
    customer_code_365      VARCHAR(50) NOT NULL,
    event                  VARCHAR(30) NOT NULL,
                                             -- 'created' / 'proposed' / 'approved' / 'rejected' /
                                             -- 'modified_by_manager' / 'cleared'
    target_weekly_ambition NUMERIC(12,2),
    target_monthly         NUMERIC(12,2),
    target_quarterly       NUMERIC(12,2),
    target_annual          NUMERIC(12,2),
    previous_weekly_ambition NUMERIC(12,2),
    previous_monthly         NUMERIC(12,2),
    previous_quarterly       NUMERIC(12,2),
    previous_annual          NUMERIC(12,2),
    actor_username         VARCHAR(64) NOT NULL,
    notes                  TEXT,
    created_at             TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_customer_spend_target_history_customer
    ON customer_spend_target_history(customer_code_365, created_at DESC);
```

**Verify column names against actual schema before applying.** `customer_code_365` and `dw_customers` are the assumed names — confirm by reading `models.py` or `dw_analytics_models.py`. If different, adjust the FK and PK accordingly.

### 6.2 — Workflow

```
                    ┌──────────────────┐
                    │ No target yet    │
                    │ (status: NULL)   │
                    └────────┬─────────┘
                             │
            AM proposes      │       Manager sets directly
        (customers.propose)  │      (customers.approve_target)
                             │
              ┌──────────────┴──────────────┐
              ▼                             ▼
    ┌──────────────────┐         ┌──────────────────┐
    │ status='proposed'│         │ status='active'  │
    │ Awaiting manager │         │ In effect        │
    └────────┬─────────┘         └────────┬─────────┘
             │                            │
             │ Manager approves           │
             │ or rejects                 │ AM proposes change
             │                            │
             ▼                            ▼
    ┌──────────────────┐         ┌──────────────────┐
    │ status='active'  │         │ status='proposed'│
    │ (or back to null)│         │ (over an active  │
    └──────────────────┘         │  target)         │
                                 └──────────────────┘
```

When an AM proposes a change to an existing active target, the active target stays in effect until the manager approves the proposal. The cockpit displays both: "Active: €80k / Proposed: €95k by maria.k, awaiting approval."

When a manager approves, the proposed values become active. The history table gets two rows: one for the proposal, one for the approval.

When a manager directly sets/changes a target without an AM proposal, status goes straight to `active` and the proposed-by fields are null. One history row recorded.

### 6.3 — Consistency hints (UI, not enforced)

In the target-edit form, when a manager fills in any of the four cadences, the others auto-populate as suggestions:

```
weekly × 52  →  annual
monthly × 12 →  annual
quarterly × 4 → annual
```

Auto-population is **suggestion only** — the form keeps each field independently editable. Real-world seasonality (HORECA accounts spike in summer) means strict consistency would be wrong. The form shows a small inline note when fields are inconsistent by more than 15%: *"Weekly × 52 = €78k, but annual is set to €95k. That's normal if you expect seasonality."*

`migrations/cockpit_002_offer_opportunity.sql`:

```sql
-- Customer-SKU pairs where:
--   1. Customer 90d revenue >= 100€ for that SKU
--   2. Customer has NO active offer for that SKU
--   3. >= 3 peer customers in same reporting group DO have an active offer for that SKU
--
-- Computed on read. If volume becomes a problem, materialize and refresh on
-- the same schedule as crm_customer_offer_current.

CREATE OR REPLACE VIEW vw_customer_offer_opportunity AS
WITH customer_sku_revenue AS (
    SELECT
        il.customer_code_365,
        il.item_code_365 AS sku,
        SUM(il.revenue_excl_vat) AS revenue_90d,
        SUM(il.gross_profit) AS gp_90d,
        AVG(il.gross_margin_pct) AS gm_pct_avg,
        MAX(il.invoice_date) AS last_bought
    FROM dw_invoice_lines il
    WHERE il.invoice_date >= CURRENT_DATE - INTERVAL '90 days'
      AND il.customer_code_365 IS NOT NULL
      AND il.item_code_365 IS NOT NULL
      AND il.revenue_excl_vat > 0
    GROUP BY il.customer_code_365, il.item_code_365
    HAVING SUM(il.revenue_excl_vat) >= 100
),
customer_active_offers AS (
    SELECT DISTINCT customer_code_365, item_code_365 AS sku
    FROM crm_customer_offer_current
    WHERE is_active = true
      AND customer_code_365 IS NOT NULL
      AND item_code_365 IS NOT NULL
),
peer_offer_penetration AS (
    SELECT
        c.reporting_group_code,
        o.item_code_365 AS sku,
        COUNT(DISTINCT o.customer_code_365) AS peer_offered_count
    FROM crm_customer_offer_current o
    JOIN dw_customers c ON c.customer_code_365 = o.customer_code_365
    WHERE o.is_active = true
      AND c.reporting_group_code IS NOT NULL
    GROUP BY c.reporting_group_code, o.item_code_365
),
customer_group AS (
    SELECT customer_code_365, reporting_group_code
    FROM dw_customers
    WHERE reporting_group_code IS NOT NULL
)
SELECT
    csr.customer_code_365,
    csr.sku,
    cg.reporting_group_code,

### 11.5 — PVM decomposition (NEW)

For the customer's sales delta vs prior year:

```
sales_delta = current_period_sales - prior_year_period_sales

For each common SKU:
  price_effect[sku]  = (current_price - prior_price) × prior_qty
  volume_effect[sku] = (current_qty - prior_qty) × prior_price

For SKUs only in current (new):
  mix_effect_new[sku] = current_revenue

For SKUs only in prior (lost):
  mix_effect_lost[sku] = -prior_revenue

price_total  = sum(price_effect.values())
volume_total = sum(volume_effect.values())
mix_total    = sum(mix_effect_new.values()) + sum(mix_effect_lost.values()) +
               cross_term_correction

sales_delta_check = price_total + volume_total + mix_total   # should equal sales_delta within rounding
```

The cross-term `(current_price - prior_price) × (current_qty - prior_qty)` for common SKUs goes into mix by convention. Document this choice in the function's docstring — finance teams sometimes argue about which bucket the cross-term belongs in.

GP-PVM uses the same logic but on `gross_profit` per line.

### 12.3 — `services/claude_advice_service.py`

```python
"""
Claude-backed advice service for the Account Manager Cockpit.

Sibling to ai_feedback_service.py (OpenAI-backed, used by the legacy
Customer Benchmark page). Two services coexist; cache table is shared
with a key prefix to prevent collision.

Greek output with English trade terminology preserved.
Focused on closing the gap-to-target.
"""
import os, json, hashlib
from datetime import datetime, timedelta, timezone

from anthropic import Anthropic
from sqlalchemy import text
from app import db

ANTHROPIC_API_KEY = os.environ.get("ANTHROPIC_API_KEY")
CLAUDE_MODEL = os.environ.get("CLAUDE_MODEL", "claude-sonnet-4-5")

CACHE_TTL_HOURS = 12
CACHE_KEY_PREFIX = "cockpit_"     # disjoint from OpenAI's (no prefix)
MAX_ROWS = 50

client = Anthropic(api_key=ANTHROPIC_API_KEY) if ANTHROPIC_API_KEY else None


GREEK_SYSTEM_PROMPT = """\
You are an experienced wholesale sales advisor for an Italian fine-foods distributor
operating in Cyprus. The user is an account manager looking at one customer's
performance and asking for actionable recommendations.

PRIMARY OPTIMIZATION SIGNAL:
The snapshot includes the customer's gap-to-target. Your recommendations must
prioritize closing that gap. Each recommendation should explicitly link to gap
reduction or risk mitigation against the target.

LANGUAGE RULES (firm, not preferences):
- Respond in Greek.
- Keep these terms in English verbatim because they are standard industry usage:
  SKU, GM%, GP, target, gap, run-rate, PVM, peer group, ABC classification,
  HORECA, Slot, ADD-ON, RFM, Index, Cross-sell, Churn Risk, White Space.
- Use Greek for verbs, reasoning, customer-friendly phrasing, action labels,
  category names where natural.
- Currency stays as €. Numbers stay as digits.

CONTENT RULES:
- Be specific. Cite exact numbers from the snapshot.
- Each recommendation must reference snapshot data (no generic platitudes).
- Prefer concrete actions ("Ανάθεση offer Q2-CHEESE στο SKU-2890").
- Prefer actions the AM can execute today (assign offer, propose order, send SMS).
- Tie expected impact to the gap. "Κλείνει €X από το €Y gap" beats "increases sales".
- If data doesn't support a recommendation, omit it. Do not pad.

OUTPUT FORMAT:
Return valid JSON with these exact top-level keys (no wrapper):
{
  "summary": "Greek - 2-3 sentence executive summary tied to the gap",
  "peer_context": "Greek - peer group description and where customer sits",
  "key_findings": ["Greek bullets, 3-5 items, each citing numbers"],
  "risks": ["Greek risk statements, 2-4 items"],
  "opportunities": [
    {"title": "Greek", "why": "Greek with English terms",
     "expected_impact": "Greek + €amount + which gap it closes",
     "confidence": 0.0-1.0}
  ],
  "next_actions": [
    {"priority": "P0|P1|P2", "action": "Greek action statement",
     "script_hint": "Greek - what to say to the customer if relevant"}
  ]
}
"""


def _hash_payload(payload: dict) -> str:
    s = json.dumps(payload, sort_keys=True, ensure_ascii=False)
    return hashlib.sha256(s.encode("utf-8")).hexdigest()


def _clip_payload(payload: dict) -> dict:
    p = dict(payload)
    for k in ["white_space", "lapsed_items", "top_items_by_gp", "top_items_by_revenue",
              "price_index_outliers", "trend", "category_mix_vs_peers",
              "active_offers", "offer_opportunities", "cross_sell",
              "churn_risk_by_category", "activity_timeline"]:
        if isinstance(p.get(k), list):
            p[k] = p[k][:MAX_ROWS]
    return p


def _cache_get(payload_hash: str):
    q = text("""
      SELECT response_json FROM ai_feedback_cache
      WHERE payload_hash = :h AND expires_at > NOW() LIMIT 1
    """)
    row = db.session.execute(q, {"h": CACHE_KEY_PREFIX + payload_hash}).fetchone()
    return row[0] if row else None


def _cache_set(payload_hash: str, response_json: dict):
    q = text("""
      INSERT INTO ai_feedback_cache(payload_hash, expires_at, response_json)
      VALUES (:h, :exp, CAST(:j AS jsonb))
      ON CONFLICT (payload_hash) DO UPDATE
      SET expires_at = EXCLUDED.expires_at,
          response_json = EXCLUDED.response_json
    """)
    exp = datetime.now(timezone.utc) + timedelta(hours=CACHE_TTL_HOURS)
    db.session.execute(q, {
        "h": CACHE_KEY_PREFIX + payload_hash, "exp": exp,
        "j": json.dumps(response_json)
    })
    db.session.commit()


def generate_cockpit_advice(snapshot: dict) -> dict:
    """
    Generate Greek-language sales advice tied to the gap-to-target.
    Mirrors ai_feedback_service.generate_feedback's contract.
    """
    if not client:
        raise ValueError(
            "Anthropic API is not configured. Set ANTHROPIC_API_KEY in Replit Secrets."
        )

    payload = _clip_payload(snapshot)
    payload_hash = _hash_payload(payload)

    cached = _cache_get(payload_hash)
    if cached:
        return cached if isinstance(cached, dict) else json.loads(cached)

    user_content = json.dumps(payload, ensure_ascii=False)

    resp = client.messages.create(
        model=CLAUDE_MODEL,
        max_tokens=4096,
        system=GREEK_SYSTEM_PROMPT,
        messages=[{"role": "user", "content": user_content}],
    )

    out_text = ""
    for block in resp.content:
        if getattr(block, "type", None) == "text":
            out_text = block.text
            break

    out_text = out_text.strip()
    # Strip markdown code fences if Claude wrapped JSON
    if out_text.startswith("```"):
        out_text = out_text.split("\n", 1)[1] if "\n" in out_text else out_text
        if out_text.endswith("```"):
            out_text = out_text.rsplit("```", 1)[0]
        out_text = out_text.strip()
        if out_text.startswith("json"):
            out_text = out_text[4:].lstrip()

    try:
        out = json.loads(out_text)
    except Exception:
        out = {
            "summary": out_text[:500], "peer_context": "",
            "key_findings": [], "risks": [],
            "opportunities": [], "next_actions": []
        }

    _cache_set(payload_hash, out)
    return out
```

### 12.4 — Endpoint

```python
@cockpit_bp.route("/api/<customer_code>/advice", methods=["POST"])
@login_required
@require_permission("customers.ask_claude")
def api_advice(customer_code):
    payload = request.get_json(silent=True) or {}
    section = payload.get("section", "all")     # 'all' | 'offers' | 'opportunities' | 'pricing' | 'risk'

    from services.cockpit_data import get_cockpit_data
    full = get_cockpit_data(customer_code)
    snapshot = _build_advice_snapshot(full, section)

    try:
        from services.claude_advice_service import generate_cockpit_advice
        out = generate_cockpit_advice(snapshot)
        return jsonify(out)
    except ValueError as e:
        return jsonify({"error": str(e), "configured": False}), 503
    except Exception as e:
        logger.exception("Claude advice generation failed")
        return jsonify({"error": "Advice generation failed", "detail": str(e)}), 500
```

`_build_advice_snapshot(full, section)` selects which keys from the full cockpit payload to include in the prompt, scoped to the section:

| section | Snapshot includes |
|---|---|
| `all` | header + target + kpis + trend + pvm + top_items_by_gp + active_offers + offer_opportunities + white_space + lapsed_items + churn_risk_by_category |
| `offers` | header + target + active_offers + offer_opportunities |
| `opportunities` | header + target + white_space + lapsed_items + cross_sell + offer_opportunities |
| `pricing` | header + price_index_outliers + top_items_by_gp |
| `risk` | header + target + churn_risk_by_category + activity_timeline + engagement |

Always include `header` and `target` because the prompt needs to know the customer and the gap.

### 12.5 — UI integration

**Recommended Actions panel** at the bottom of the cockpit fetches `section="all"` on page load. Renders the top 3–4 `next_actions` from the response. If the cache has a hit (cheap), render synchronously in the server-rendered template. If not, render a loading skeleton and fetch async after page paint.

If the user lacks `customers.ask_claude`, hide the Recommended Actions panel entirely (no fallback content — managers without permission can still use the rest of the cockpit, just without AI advice).

**Per-section "✦ Ask Claude" buttons** appear at the top of each major card (Buying, Offers, Opportunities, Pricing, Risk). Each button calls `askClaude(section)` which posts to the endpoint with the corresponding section key and renders the response in a modal.

**Button gating:** show buttons only if `has_permission('customers.ask_claude')`. The page-level button reads "✦ Ask Claude" — section-level buttons read "✦ Ask Claude about offers" etc.

### 12.6 — Failure handling

If `ANTHROPIC_API_KEY` is unset → endpoint returns 503 with `{"configured": false}`. Cockpit renders fully, Recommended Actions panel shows: *"Συμβουλές μη διαθέσιμες — επικοινώνησε με admin."* No 500.

If Anthropic API returns an error (rate limit, timeout, etc.) → 500 caught, panel shows: *"Σφάλμα κατά τη δημιουργία συμβουλής. Δοκίμασε ξανά."* Full error logged server-side.
