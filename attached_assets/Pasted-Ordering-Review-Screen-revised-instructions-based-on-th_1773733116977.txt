Ordering Review Screen — revised instructions based on the current page

We are keeping the current flat queue layout on the Ordering Review screen.
Do not add accordion expansion here.

The goal is to make this page a fast operational review queue with compact rows and a right-side detail drawer when a row is clicked.

1) Page purpose

This screen must be only for reviewing customers currently in the ordering window.

It should help the operator answer quickly:

who needs review now

why they need review

what action to take

whether the cycle is closed

It should not behave like a full CRM/customer profile page.

2) Keep the table flat

Do not add row expansion.

Required behavior:

each row stays compact

clicking a row opens a right-side drawer

the drawer shows the detailed review info and actions

the main list remains fast to scan and scroll

3) Simplify the state model

Replace the current broad PENDING / DONE style with these 5 review states only:

waiting

follow_up

ordered

ordered_cart

done

Meaning

waiting
Customer is in the ordering window, no order yet, no special follow-up trigger yet.

follow_up
Customer needs review/action now.

ordered
Order exists, no extra issue.

ordered_cart
Order exists and there is still an active cart.

done
Operator has reviewed and closed this cycle for this customer.

4) State logic — mostly automatic

States should mostly be set automatically by the system.

Automatic rules

Apply in this order:

Rule 1
If operator marked the record done → done

Rule 2
Else if order exists and active cart exists → ordered_cart

Rule 3
Else if order exists → ordered

Rule 4
Else if assisted-ordering flag is true and no order exists → follow_up

Rule 5
Else if active cart exists and no order exists → follow_up

Rule 6
Else if customer logged in during the current ordering window and still no order exists → follow_up

Rule 7
Else if operator manually marked expected_this_cycle = true and no order exists → follow_up

Rule 8
Else → waiting

Operator controls

Operator should only be able to:

manually mark follow_up

manually mark done

reopen a done row back to follow_up

set outcome reason when done

optional short notes

Assisted ordering — how to implement

This is a customer-level operational flag, not a per-cycle review field.

Implement it as a boolean field on the customer CRM/profile record, for example in CrmCustomerProfile:

assisted_ordering boolean not null default false

This flag means: this customer is known to usually need human assistance/follow-up to place orders.

Use it in two ways:

in the ordering review state logic, where assisted customers move to follow_up earlier

as a filter on the ordering screen (Assisted only)

For phase 1:

display this field in the drawer

allow it to be edited in the drawer with a simple checkbox/toggle and save button

This field should not be stored in the ordering review record, because it is a persistent customer behavior, not a cycle-specific event.

Everything else should be handled later inside the drawer.

12) Data fields needed

Split the fields into customer-level and ordering-review-level.

A. Customer-level field

This belongs to the CRM/profile record, not the review record:

Add or confirm these fields for the ordering review record:

13) Important business behavior to preserve

Please keep the business rules below exactly:

biweekly is already handled by the ordering-window engine, so no extra UI treatment is needed

some customers are assisted-ordering customers and should move to follow_up earlier

some customers may legitimately skip a cycle, so this should be captured as an outcome reason (valid_skip), not as a permanent negative state

active cart with no order means operator should check whether customer thinks the order was already sent or help them close the order

active cart after an existing order means operator should check whether they want the extra items added

14) Final implementation direction

Build the Ordering Review screen as:

a compact flat queue

no accordion

row click opens right-side detail drawer

5 simple states only

most states set automatically

operator only confirms follow-up / done / outcome

cart meaning split into pending vs add-on

minimal inline actions

detailed actions progressively added inside the drawer

for now, only Send SMS is implemented in Quick actions

action log is deferred to a later phase

assisted ordering is implemented as a customer-level boolean flag and shown/edited in the drawer

This will make the screen faster, cleaner, and much better aligned with the actual ordering-window process.
