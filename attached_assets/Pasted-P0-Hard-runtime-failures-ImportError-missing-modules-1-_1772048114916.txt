5) services_discrepancy.get_route_settlement_summary() is broken SQL + wrong status check

File: services_discrepancy.py

line 297 joins payment_customers pt ON pt.customer_code_365 = i.customer_code_365
✅ but PaymentCustomer model uses column code, not customer_code_365.

line 292 selects pt.payment_type_code_365
✅ but PaymentCustomer model has no payment_type_code_365.

line 331 checks if row.delivery_status == 'DELIVERED': (case-sensitive)
✅ but elsewhere you normalize to lowercase (and delivery_status.py canonicalizes to lowercase).

Replace payment_customers join/fields with credit_terms or invoice_payment_expectations / cod_invoice_allocations.

Normalize status in SQL: LOWER(rsi.status) = 'delivered'.

P1 — Data correctness bugs (will make routing/reconciliation wrong)
6) Invoice sync does not populate customer code fields

File: services_powersoft.py (invoice sync)

In the invoice upsert (~205–211), you set:

invoice_no, customer_name, upload_date, status, total_grand

You do not set Invoice.customer_code_365 and Invoice.customer_code (even though PS365 header likely provides it)

Impact: anything relying on customer code (routing/grouping, credit terms lookup, reconciliation joins) will fail or fall back to fuzzy matching by name.

Fix: when parsing PS365 header, set:

invoice_record.customer_code_365 = header.get("customer_code_365")

Decide one canonical field (customer_code_365 vs customer_code) and consistently use it everywhere (right now you use both across the system).

7) Credit terms lookup in reconciliation uses i.customer_code (likely empty)

File: services_reconciliation.py

In get_invoice_reconciliation_report() lateral join:

WHERE ct2.customer_code = i.customer_code

Impact: reconciliation “POD vs CREDIT” logic becomes wrong if i.customer_code isn’t filled (and per #6 it usually won’t be).

Fix: use the canonical customer code field you actually populate (likely i.customer_code_365) and make sure it matches credit_terms.customer_code (which references payment_customers.code).

P1 — Financial precision + idempotency issues
8) Money stored as floats in multiple places (rounding drift risk)

services_powersoft.py:198 total_grand_value = float(...) stored into Invoice.total_grand (Numeric)

routes_receipts.py passes amount_val: float and stores it into ReceiptLog.amount (Numeric)

Impact: subtle cent-level rounding errors over time.

Fix: parse amounts as Decimal, quantize to 2dp, store Decimal.

9) Receipt reference sequence can be reused after rollback (bad idempotency)

File: routes_receipts.py:55–78 next_reference_number()

Because the “last_number” update is inside the same DB transaction as the external API call + logging, a rollback can revert the sequence increment. If PS365 actually created the receipt but your DB rolled back, you can reuse the same reference number later.

Fix: use a Postgres SEQUENCE (nextval() is not rolled back), or increment the sequence row in an autocommit context before calling PS365.

P1 — Functional bugs in routing logic

11) Cleanup after reroute will crash (missing services.delete_stop)

File: services_routing.py:279

P3 — Structural/maintainability issues that cause real bugs later
15) Too much schema mutation at import/startup (multi-worker race risk)

app.py runs db.create_all() and schema updates during import/app context.

main.py runs a long list of schema updates again.

Impact: under gunicorn multi-worker, each worker can race migrations, cause locks, slow boot, and intermittent startup failures.

Fix: move schema migrations to a controlled step (Alembic), or at least ensure they run once (leader lock table, etc.).
