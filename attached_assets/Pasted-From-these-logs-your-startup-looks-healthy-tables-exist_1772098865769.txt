Production-safe fix: ensure ONLY ONE process runs jobs (DB advisory lock)

This prevents duplicate jobs even if you later run 2–4 workers.

# app/scheduler_lock.py
from sqlalchemy import text

_lock_conn = None

def try_acquire_scheduler_lock(engine, lock_name="wmds_scheduler"):
    global _lock_conn
    if _lock_conn is not None:
        return True  # already acquired in this process

    conn = engine.connect()  # keep open to hold lock
    got = conn.execute(
        text("SELECT pg_try_advisory_lock(hashtext(:name))"),
        {"name": lock_name},
    ).scalar()

    if got:
        _lock_conn = conn
        return True

    conn.close()
    return False
# in create_app()
from app.scheduler_singleton import get_scheduler
from app.scheduler_lock import try_acquire_scheduler_lock
from app import db

def init_scheduler(app):
    if not try_acquire_scheduler_lock(db.engine):
        app.logger.info("Scheduler lock not acquired; not starting in this process")
        return

    sched = get_scheduler()
    if not getattr(sched, "running", False):
        sched.start()
        app.logger.info("Background scheduler started (lock acquired)")

Why this matters: if you ever add jobs (syncs, alerts, retries), you’ll otherwise run them twice and create duplicate rows / duplicate emails / duplicate “alerts”.

If Replit is “crashing” (the process restarts), it’s almost always one of these three buckets:

Worker is being killed (timeout or OOM/memory)

Unhandled exception (usually during startup or in a background job)

Run-loop duplication (scheduler/threads start multiple times → leaks / duplicate jobs / runaway CPU)

Your earlier log already shows duplicate APScheduler startups, so bucket #3 is a prime suspect — and it can also cause #1 (memory/CPU growth) over time.

Below is the fastest way to pinpoint + fix without guessing.

A. Identify which “crash” it is (look for ONE of these signatures)

Open Replit Console → scroll to the last 30 lines before it restarts and look for:

1) Memory kill (most common on Replit)

Clues:

Killed

OOM

exit code 137

sudden stop with no traceback

Fix pattern: stop duplicate schedulers, reduce in-memory loads, add pagination, avoid building huge lists/JSON, don’t preload entire invoices/routes, etc.

2) Gunicorn timeout kill

Clues:

WORKER TIMEOUT

Worker exiting (pid ...)

exit code 1 after timeout messages

Fix pattern: any request that can take >300s must be moved off-request (background job + progress), or you must raise timeout (but that’s usually the wrong fix long-term).

3) Real Python crash (traceback)

Clues:

stack trace ending in Exception: / Traceback

Fix pattern: fix the specific exception (often missing app context in background thread/job, DB disconnect handling, None access, etc.).

B. Fix the “scheduler starts twice” issue now (this alone can stop the crashes)

Your log shows Scheduler started more than once. In Replit, the app can be imported multiple times (reloads, multiple entrypoints, gunicorn preloading, etc.). If each import starts APScheduler, you get multiple scheduler threads and eventually instability.

Make scheduler start lock-guarded (production-safe)

This guarantees only one process/thread actually runs scheduler, even if the app is imported multiple times or you scale workers later.

# scheduler_guard.py
from sqlalchemy import text

_lock_conn = None

def acquire_lock(engine, name="wmds_scheduler"):
    global _lock_conn
    if _lock_conn is not None:
        return True

    conn = engine.connect()  # keep open to hold lock
    got = conn.execute(
        text("SELECT pg_try_advisory_lock(hashtext(:n))"),
        {"n": name},
    ).scalar()

    if got:
        _lock_conn = conn
        return True

    conn.close()
    return False
# init_scheduler.py
from apscheduler.schedulers.background import BackgroundScheduler
from scheduler_guard import acquire_lock

_scheduler = None

def init_scheduler(app, engine):
    global _scheduler
    if not acquire_lock(engine):
        app.logger.info("Scheduler lock NOT acquired; skipping scheduler start.")
        return

    if _scheduler and getattr(_scheduler, "running", False):
        app.logger.info("Scheduler already running; skip.")
        return

    _scheduler = BackgroundScheduler(timezone="Asia/Nicosia")
    _scheduler.start()
    app.logger.info("Scheduler started (lock acquired).")

Then call init_scheduler(app, db.engine) in one place only (app factory is ideal).

import os, time, threading
try:
    import psutil
except ImportError:
    psutil = None

def start_resource_logger(app):
    if not psutil:
        app.logger.info("psutil not installed; resource logger disabled")
        return

    p = psutil.Process(os.getpid())

    def loop():
        while True:
            rss = p.memory_info().rss / (1024 * 1024)
            cpu = p.cpu_percent(interval=1.0)
            app.logger.info(f"[RES] rss={rss:.1f}MB cpu={cpu:.1f}% pid={p.pid}")
            time.sleep(30)

    t = threading.Thread(target=loop, daemon=True)
    t.start()

If you see RSS climbing steadily until it dies → OOM.
If RSS stable but crashes during specific actions → timeout or exception.
