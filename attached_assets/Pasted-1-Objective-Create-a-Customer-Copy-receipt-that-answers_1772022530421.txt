1) Objective

Create a Customer Copy receipt that answers only:

Who collected (your company + collector/driver)

From whom (customer)

When (date/time)

What was collected (amount + payment method breakdown)

Which invoices it relates to (invoice numbers only)

Signature (customer)

Remove operational/internal noise (route/stop/cust code/exceptions/PS365 refs).

2) Receipt content rules (Customer Copy)
Keep

Company name + VAT + phone

Title: COLLECTION RECEIPT (optionally add Greek: ΑΠΟΔΕΙΞΗ ΕΙΣΠΡΑΞΗΣ)

Receipt No

Date + Time

Customer name (address optional)

Invoice references: invoice numbers only

Payment method breakdown (Cash / Cheque / Card / Bank) + amounts

Total collected

Collector name

Customer signature line

Remove

“STATUS: PAID” (do not show status at all)

PS365 Ref

Route / Stop

Customer code

Invoice amounts and invoice totals

Balance due

Exceptions / SHORT / Exp/Act etc.

This avoids disputes: the receipt becomes purely an acknowledgement of the amount received.

3) Print layout (58mm-friendly)

Use a monospace layout and keep lines short (≈ 32 chars). Suggested output:

STEP EPLATTFORMA LTD

Date: 2026-02-25  Time: 09:15

(autowrap to next lines)

Collected:
Cash:   132.22
Total:  132.22 EUR

Collector: Ricardo

Customer Signature: ____________

Important: invoices list must wrap cleanly to multiple lines if many.

4) Backend data requirements (what the print needs)

When generating the receipt, build a data object like:

receipt_no (string)

collected_at (datetime)

customer_name (string)

customer_address (optional string)

invoice_nos (list of strings) ✅ only these

payments (list of {method, amount}) e.g. CASH 132.22

total_collected (sum of payments)

collector_name (driver/user display name)

currency (EUR)

No balances, no invoice totals.

6) New receipt template (Customer Copy)
A) Create a dedicated template

templates/receipts/collection_receipt_customer.html (or .txt if you print as raw text)

Must render only the fields above.

B) Wrap invoice numbers

Implement a helper that wraps a long comma-separated invoice list to printer width.

Example approach (Python helper used before rendering):

def wrap_invoice_refs(invoice_nos, width=32):
    s = ", ".join(invoice_nos)
    lines, line = [], ""
    for part in s.split(", "):
        add = part if not line else ", " + part
        if len(line) + len(add) > width:
            lines.append(line)
            line = part
        else:
            line += add
    if line:
        lines.append(line)
    return lines

Then in the template print each line separately.

7) Print endpoint / route changes
A) Add a “Customer Copy” print option

Wherever you currently print the existing receipt, add:

copy=customer (default)

copy=internal (optional later if you still want the old details for your office)

Suggested endpoint pattern:

GET /receipts/<receipt_id>/print?copy=customer&format=raw

or format=pdf if you must use PDF.

B) Build the render payload

In the endpoint:

Load receipt header

Load invoice refs list

Load payment lines list

Compute total_collected = sum(amount)

Create invoice_ref_lines = wrap_invoice_refs(invoice_nos)

Render template with payload

Send to printer

8) UI changes (Driver app)

On “Payment collected / Closeout” screen:

Keep the same workflow

Replace the print action label to:

“Print Collection Receipt”

After saving collection, show:

Receipt No

Total collected

Print button

No need to show balances to the driver if you don’t trust them.

9) Acceptance tests (must pass)

Single cash payment → prints Cash + Total, invoices list

Multiple payment methods (cash + cheque) → prints both lines + correct total

Many invoices (10–30 refs) → invoice numbers wrap neatly (no overflow)

Receipt never prints:

invoice amounts

totals invoiced

balance due

exceptions

route/stop/cust code/PS365 ref

Works on both Android + iOS print paths (same receipt content)

10) Optional (strongly recommended)

Print footer line:

“Payment acknowledgement only.”
(or bilingual if you want)

This reduces “is this a tax receipt?” confusion.
