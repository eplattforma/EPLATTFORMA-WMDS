Bug Fix: Lock Cooler Sequencing Returns stamped: 0
What's Happening
Clicking "Lock cooler sequencing" on the cooler picking screen returns:
json{
  "locked_at": "2026-05-12T12:07:34.428102+00:00",
  "locked_by": "Polis",
  "ok": true,
  "route_id": 409,
  "session_id": 636,
  "skipped_no_stop": 0,
  "stamped": 0
}
stamped: 0 means the lock ran but found zero items to stamp with a delivery sequence. Items remain in "Unsequenced — awaiting lock" and cannot be picked.
Root Cause To Investigate
The lock query joins batch_pick_queue → route_stop_invoice → route_stop to get seq_no. stamped: 0 means this join returns nothing — the cooler queue rows exist but cannot resolve their stop sequence.
Two likely causes:
Cause A — The cooler session (COOLER-ROUTE-409, session 636) was created through the old batch picking path, not through the Phase 6 extract_sensitive_for_route_stop_invoices() hook. The queue rows have invoice_no populated but the route_stop_invoice table has no active row linking those invoices to their stops.
Cause B — The invoices are correctly in route_stop_invoice but is_active = false on those rows, causing the join to miss them.

What To Fix

Find the lock cooler sequencing endpoint in blueprints/cooler_picking.py. Currently when seq_no cannot be resolved it silently skips (hence skipped_no_stop: 0 matching the count, and stamped: 0).

Fix 2 — The lock must also search by route_id not just batch_session_id
The lock query likely filters cooler queue rows by batch_session_id. But cooler sessions created via the old batch picking path (COOLER-ROUTE-N naming) may have a different session structure than Phase 6 expects.
Check whether the lock endpoint finds items by batch_session_id OR by route_id via the invoice join. It should use both as fallbacks:

Fix 3 — If route_stop_invoice rows are missing for these invoices
If Q2 returns zero rows, the invoices in the cooler queue are not linked to route stops. The Phase 6 extract_sensitive_for_route_stop_invoices() hook needs to be triggered manually for these existing sessions.
Add a "Re-extract cooler items" button on the cooler picking screen that re-runs the extraction for all invoices currently on the route. This is the idempotent re-run path that already exists in extract_sensitive_for_route_stop_invoices() — just expose it as a UI action for cases where extraction was missed.
