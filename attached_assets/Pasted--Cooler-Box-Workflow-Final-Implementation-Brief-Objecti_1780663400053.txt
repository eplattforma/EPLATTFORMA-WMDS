# Cooler Box Workflow Final Implementation Brief

## Objective

Simplify and finalise the cooler box process as a **warehouse operation**, not a delivery-route operation.

The system should support this operational flow:

1. Warehouse selects a cooler route/date.
2. Warehouse confirms the delivery sequence.
3. Warehouse plans cooler boxes using the delivery stop order.
4. Box 1 contains the **last delivery stops**.
5. Picker starts picking Box 1 and continues until Box 1 is complete.
6. When Box 1 is complete, the picker gets a clear message to close/seal Box 1 and start Box 2.
7. Repeat until all boxes are picked and closed.
8. After completion, the route becomes **Cooler Ready for Dispatch**.
9. Warehouse must be able to access completed cooler boxes from the Warehouse menu, not only from Delivery Route.


# 6. Box-first picking logic

The picker must not pick by general location-first order across all boxes.

For cooler batches, picking sequence must follow confirmed cooler boxes:

```text
Box 1
Box 1
Box 1
Box 1 complete
Box 2
Box 2
Box 2 complete
Box 3
...
```

Box 1 contains the last delivery stops.

Inside each box, items may be sorted by warehouse location.

Required order:

```text
box_no ASC
then warehouse location order inside the box
```

Do not consolidate items across different boxes.

Grouping key must include:

```text
box_id
item_code
location
```

Do not group only by:

```text
item_code
location
```

For `session_type = 'cooler_route'`, build picker items from confirmed:

```text
cooler_boxes
cooler_box_items
batch_pick_queue
```

not from the normal location-only batch list.

Use or improve existing helper:

```python
build_cooler_box_picking_items(batch_session)
```

Each picker item should include:

```text
box_id
box_no
box_type_name
box_status
box_stop_display
item_index_in_box
total_items_in_box
is_first_item_in_box
is_last_item_in_box
next_box_no
next_box_type_name
invoice_no
item_code
item_name
location
qty
barcode
```

Before starting cooler picking, require a confirmed box plan.

If no confirmed boxes exist, block Start Picking and redirect user to the route packing screen to plan boxes first.

# 15. Acceptance criteria

Implementation is complete when all of the following are true:

1. Warehouse menu has a permanent Cooler Packing / Cooler Boxes Handover entry.
2. Completed closed cooler boxes can be accessed without going through Delivery Route.
3. `/cooler/route-list` shows active and completed cooler routes.
4. `/cooler/route/<route_id>/<delivery_date>` is visually a warehouse cooler control/handover page.
5. Top KPIs are always visible.
6. Middle Next Action card shows only the relevant next action.
7. Route Item Status table is available and printable.
8. Customer Code and Queue ID are not shown in normal Route Item Status.
9. Normal box changes happen only inside Box Planning before confirmation.
10. Advanced Box Adjustments is renamed to Manager Recovery Tools and hidden/collapsed/admin-only.
11. Picker follows Box 1 completely, then Box 2, then Box 3.
12. Picker gets a clear modal/message when starting a new box.
13. Picker gets a clear modal/message when a box is complete and must be closed.
14. Single-item boxes show one clear message, not multiple stacked banners.
15. After all boxes are closed, page shows Cooler Ready for Dispatch.
16. Final screen shows Closed Boxes / Dispatch Handover expanded by default.
17. Final handover table shows box status, stops, items, fill, closed time, label status and actions.
18. Print All Box Labels exists or is clearly implemented.
19. Route Manifest and Route Status print remain available.
20. Delivery Route is not the only way to find cooler boxes.
