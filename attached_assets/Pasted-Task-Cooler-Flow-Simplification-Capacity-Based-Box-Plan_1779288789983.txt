PART A — COOLER FLOW SIMPLIFICATION
===================================

Current problem:
The cooler picking screen mixes route preparation, picking method, manual assignment, and box packing into one complicated interface.

Correct business process:

1. Prepare Cooler Route.
2. Picker picks all cooler items using the normal batch picking interface, sorted by warehouse location.
3. After picking, the system generates a box plan.
4. Box plan groups multiple delivery stops into cooler boxes based on capacity.
5. Manager confirms the box plan.
6. Boxes are created as open.
7. User closes boxes only when physically sealed.
8. Labels/manifest can then be printed.
9. Cooler route/session completes only when all cooler items are picked, boxed, and boxes are closed.

Do not use “Pack by Stop” as the main workflow.
Do not create one box per stop as the normal process.

Correct example:

* Box 1 = Stops 10 → 8
* Box 2 = Stops 7 → 5
* Box 3 = Stops 4 → 1

Wrong default process:

* Box 1 = Stop 10
* Box 2 = Stop 9
* Box 3 = Stop 8

A cooler box is a physical container. It may contain multiple stops.

## A4. Create new service: `services/cooler_box_planner.py`

Create:

`services/cooler_box_planner.py`

Add function:

`generate_box_plan(route_id, delivery_date, box_type_id=None)`

The function must generate a proposed box plan from picked, unboxed cooler items.

Use the existing data structure already used in `route_picking(route_id, delivery_date)`:

`batch_pick_queue`
joined to:

* `invoices`
* `shipments`
* `route_stop_invoice`
* `route_stop`
* `invoice_items`
* `ps_items_dw`

Required filters:

* `batch_pick_queue.pick_zone_type = 'cooler'`
* `batch_pick_queue.status = 'picked'`
* `invoices.route_id = route_id`
* `shipments.delivery_date = delivery_date`
* exclude queue rows already assigned in `cooler_box_items.queue_item_id`

Required sorting:

* delivery stop sequence descending, so higher stop numbers are planned first.

Required grouping:

* group items by delivery stop first;
* keep one stop together where possible;
* allow multiple stops in one box;
* do not create one box per stop unless capacity requires it.

Box capacity:
Use `cooler_box_types`.

For selected box type:

* use selected `box_type_id`.

If no box type is selected:

* use an active box type from `cooler_box_types`;
* choose the largest active usable volume by default.

Usable capacity:
`internal_volume_cm3 * fill_efficiency`

Item volume:
`item_length * item_width * item_height * qty`

Item weight:
`item_weight * qty`

Dimension source:
`ps_items_dw` / `DwItem`.

Missing dimensions:
If item length, width, or height is missing:

* do not crash;
* include the item in the plan;
* increase `missing_dimension_count`;
* include a warning that the box fill estimate is incomplete.

Required return structure per planned box:

        {
            "queue_item_id": 101,

Display stop direction as:
`Stops 10 → 8`

`Stops 8–10`

Reason:
The physical loading/packing logic is last stop first.
