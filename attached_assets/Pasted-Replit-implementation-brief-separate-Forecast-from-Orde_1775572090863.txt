Replit implementation brief — separate Forecast from Ordering, keep OOS-aware forecasting, add per-item Weeks of Stock
Objective

Refactor the current forecast pipeline so that:

Forecast run only computes demand forecast
weekly sales
seasonality
classification
OOS-aware base/final forecast
forecast metadata
Ordering refresh becomes a separate process
fetch latest stock / reserved / incoming
use latest stored forecast
compute target stock and suggested order
can be run manually on demand
Add a persistent per-item target weeks of stock parameter
default 4.0
editable by user
remains unchanged until user changes it again
Preserve and improve the current OOS adjustment logic in forecasting.

8) Add API endpoint to update per-item weeks of stock

Still in blueprints/forecast_workbench.py, add:

@forecast_bp.route('/api/item/<item_code>/target-weeks', methods=['POST'])
@admin_or_warehouse_required
def api_update_target_weeks(item_code):
    data = request.get_json(silent=True) or request.form
    val = data.get('target_weeks_of_stock')

    try:
        target_weeks = float(val)
    except (TypeError, ValueError):
        return jsonify({"error": "Invalid target_weeks_of_stock"}), 400

    if target_weeks < 0:
        return jsonify({"error": "target_weeks_of_stock must be >= 0"}), 400

    profile = SkuForecastProfile.query.get(item_code)
    if not profile:
        profile = SkuForecastProfile(item_code_365=item_code)
        db.session.add(profile)

    profile.target_weeks_of_stock = Decimal(str(round(target_weeks, 4)))
    profile.target_weeks_updated_at = get_utc_now()
    profile.target_weeks_updated_by = current_user.username

    db.session.commit()
    return jsonify({
        "status": "ok",
        "item_code": item_code,
        "target_weeks_of_stock": float(profile.target_weeks_of_stock)
    })

This must persist permanently until changed again.

16) Acceptance criteria

Implementation is correct only if all of the following are true:

Clicking Run Forecast does not fetch live stock and does not compute suggested order.
forecast_runs.current_step ends at base_forecast then completed.
SkuForecastProfile.target_weeks_of_stock exists and defaults to 4.0.
Editing target weeks on an item persists and remains after next forecast run.
Clicking Refresh Ordering fetches latest stock and writes rows to sku_ordering_snapshot.
Suggested order in UI comes from latest ordering snapshot, not sku_forecast_result.
OOS adjustment still affects classification and forecasting.
Trend is suppressed when recent calendar weeks are OOS-impacted.
Forecast screens still show weekly forecast, trend, seasonality, confidence, OOS markers.
Ordering can be refreshed repeatedly without rerunning forecast.
