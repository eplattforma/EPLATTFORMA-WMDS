Replit implementation instruction — replace replenishment logic with date-independent 7-day stock cover

We need to change the replenishment engine, not just the forecast display.

Business rule

Replenishment must ignore calendar date patterns and simply maintain approximately 7 days of stock forward.

This means:

do not forecast based on the weekdays of the next 7 dates

do not use “same weekday averages” as the main replenishment driver

do not depend on a run type if replenishment now follows one fixed policy

1) Remove old weekday-based replenishment logic

Please remove the current replenishment logic that:

identifies the weekdays in the next 7-day horizon

looks back to the last 4 same weekdays

averages them

sums those weekday averages to produce a 7-day forecast

That logic is not aligned with the new business rule.

Keep any historical code only if needed for audit, but it must not drive replenishment anymore.

2) Replace it with weekly-demand-based replenishment

Build replenishment from recent weekly demand history.

Required basis

use real invoice dates from invoice header

aggregate demand into full weekly buckets per SKU

fill missing weeks with zero

use the last 26 weeks for classification

use gross sales demand as the main demand signal

keep returns/credits separate for diagnostics, not as the main forecast driver

Rules

if no sales in last 26 weeks → no_demand

if only very limited history / too few non-zero weeks → new_sparse

otherwise use ADI and CV²:

ADI < 1.32 and CV² < 0.49 → smooth

ADI < 1.32 and CV² >= 0.49 → erratic

ADI >= 1.32 and CV² < 0.49 → intermittent

ADI >= 1.32 and CV² >= 0.49 → lumpy

4) Base forecast method by demand class
Smooth

Use:

MA8 = average of last 8 weekly quantities

Intermittent / Lumpy

Use:

Median6 = median of last 6 weekly quantities

Erratic

For now, use:

MA8
but set review flag more aggressively

No demand

Use:

weekly_forecast = 0
and flag for review if stock exists or item is active

New / sparse

This is important.

If an item has only 1 week sales or very little history, do not trust its own history alone.

Classify it as:

new_sparse

low confidence

review required

Use this fallback logic:

Preferred method

If a good analogue exists:

same supplier if available

else same brand

else same prefix/category

Then use:

seeded_weekly_forecast = 0.35 × first_week_sales + 0.65 × analogue_baseline

If only weak analogue exists

Use:

seeded_weekly_forecast = 0.50 × first_week_sales + 0.50 × brand/prefix baseline

If no analogue exists

Use:

seeded_weekly_forecast = first_week_sales × 0.50

Do not use MA8 for items with only 1 week sales.

Store:

forecast_method = SEEDED

forecast_confidence = low

review_flag = true

seed_source = supplier/brand/prefix/none

5) Convert weekly forecast to daily forecast

After selecting the weekly forecast:

daily_forecast = weekly_forecast / 7

This is the base for replenishment.

6) Trend handling

For smooth items only, add a controlled recent trend adjustment.

Calculate:

MA8

AvgLast2 = average of last 2 weekly quantities

Uptrend rule

If:

AvgLast2 > MA8 × 1.15

Then:

trend_flag = up

trend_adjusted_weekly = min(MA8 × 1.25, MA8 + 0.5 × (AvgLast2 - MA8))

Downtrend rule

If:

both of last 2 weeks are below MA8

and AvgLast2 < MA8 × 0.90

Then:

trend_flag = down

trend_adjusted_weekly = max(MA8 × 0.75, MA8 - 0.5 × (MA8 - AvgLast2))

Else:

trend_flag = flat

trend_adjusted_weekly = MA8

For non-smooth items in phase 1:

trend_adjusted_weekly = base_weekly_forecast

7) Replenishment formula
Fixed replenishment policy

The replenishment engine should now follow one policy:

hold about 7 days of stock forward

Use:

target_stock_qty = daily_forecast × (cover_days + lead_time_days + review_cycle_days) + safety_or_buffer_qty

Then:

net_available_qty = on_hand_qty + incoming_qty - reserved_qty

Then:

raw_recommended_order_qty = max(0, target_stock_qty - net_available_qty)

Then round:

respect MOQ

respect case qty

respect order multiple if configured

Return:

rounded_order_qty

Defaults

cover_days = 7

review_cycle_days = 1

lead_time_days from supplier/item settings, else default 0 until configured

8) Safety stock / buffer stock

Please check the current calculation carefully.

For SNA-0118, the previous screen showed:

forecast for 7 days = 56.73

safety stock = 8.1

Since 56.73 / 7 ≈ 8.1, it looks like the current “safety stock” may just be 1 day of forecast, not true safety stock.

Required action

Please verify exactly how this field is calculated.

If it is just one extra day of demand, then:

either rename it to buffer_stock_qty

or implement a true statistical safety stock formula later

For now, honesty and clarity are more important than pretending it is statistical safety stock.

My preference for phase 1:

use buffer_stock_qty

and make it configurable

default it to either 0 or 1 day of demand, but label it correctly

Do not call it statistical safety stock unless it is actually calculated as one.

12) Acceptance criteria

This change is complete when:

Replenishment no longer uses weekday-based next-date forecasting.

Replenishment uses weekly demand history with zero-filled weeks.

Smooth items use MA8.

Intermittent/lumpy items use Median6.

New items with only 1 week sales use SEEDED logic, not MA8.

Daily forecast is derived from weekly forecast.

Target stock is based on 7-day stock cover, not calendar weekday logic.

MOQ/case/order multiple rounding works correctly.

The field called safety stock is either proven correct or renamed to buffer stock.

The UI clearly explains how each order quantity was produced.

SNA-0118 can be debugged with a full intermediate-value breakdown.
