Replit implementation brief — Supplier Forecast & Order Workbench

Build this in our existing Flask + SQLAlchemy + PostgreSQL app.

The goal is to create a supplier-first forecasting and replenishment workbench that uses:

dw_invoice_header

dw_invoice_line

ps_items_dw

current stock / reserved / incoming sources already available in the app where possible

The workbench must let us review forecast by supplier first, then drill into category / brand / SKU, and clearly show:

whether an SKU is smooth / erratic / intermittent / lumpy / new-sparse / no-demand

the forecast method used

whether there is recent trend up/down

whether there is a seasonality layer

the recommended order quantity

item master data relevant to ordering such as MOQ, case qty, active flag, supplier item code

This is not only a reporting page. It must produce a usable ordering view by supplier.

1. Architecture to implement

Keep raw tables unchanged:

dw_invoice_header

dw_invoice_line

ps_items_dw

Add a forecasting/reporting layer on top.

Implement these layers:

A. Weekly sales fact

A materialized table for weekly item demand.

B. SKU forecast profile

A table storing demand classification, method, trend, seasonality source, confidence.

C. SKU forecast result

A table storing the final weekly forecast and replenishment outputs.

D. Supplier forecast view

A supplier-first UI that reads from the forecast result/profile plus item master plus stock.

2. Important business rules
2.1 Supplier-first workflow

The primary planning workflow is:

Supplier → Category / Brand → SKU

Do not make SKU list the main landing page.
Landing page must be supplier-focused.

2.2 Sales basis

Use real invoice dates from dw_invoice_header joined to invoice lines by invoice_no_365.

Do not use sync timestamp for forecasting.

2.3 Time grain

Forecasting must be based on weekly buckets, not daily raw lines.

2.4 Demand source

Use gross sold qty from invoice-type sales as base demand.

Returns / credit notes must be stored separately and shown as diagnostics, not blindly mixed into demand signal.

2.5 Classification

Classify each SKU based on the last 26 weeks with zero-filled missing weeks.

2.6 Forecast logic

Default logic:

Smooth → MA8

Erratic → MA8 with stronger review flag, or blended logic later

Intermittent / Lumpy → Median6

New/Sparse → fallback seeded/simple logic

No-demand → zero / manual review

2.7 Trend logic

For smooth/regular SKUs, allow short-term trend adjustment.

Base idea:

base forecast = MA8

calculate AvgLast2

if AvgLast2 materially above or below MA8, apply controlled capped uplift/downside adjustment

2.8 Seasonality logic

Use the ratio-based method we discussed.

Do not multiply MA8 by just the current month factor.

Use:

Final Forecast = Base Forecast × (Future Seasonal Index / Historical Embedded Seasonal Index)

Where:

Historical Embedded Seasonal Index = weighted average seasonal factor of the last 8 weeks used by MA8

Future Seasonal Index = weighted average seasonal factor of the forward cover horizon

For now, seasonality should be primarily from:

prefix/family level

optionally brand level if enough volume

SKU-level only if history is strong enough

2.9 Item master vs statistical seasonality

Keep these separate in the UI:

season_code_365 from item master

forecast seasonality signal from sales pattern

They are not the same thing.

5. Forecast calculations
5.1 Classification metrics

For each SKU over last 26 complete weeks:

weeks_total_26 = 26

weeks_non_zero_26

sales_frequency_26 = weeks_non_zero_26 / 26

ADI = 26 / weeks_non_zero_26 if non-zero weeks > 0

compute average and std deviation on non-zero weekly quantities

CV² = (std_non_zero / avg_non_zero)^2 where average > 0

Classification rules:

if weeks_non_zero_26 = 0 → no_demand

if insufficient history / very sparse, e.g. < 6 non-zero weeks or very recent intro → new_sparse

else:

ADI < 1.32 and CV² < 0.49 → smooth

ADI < 1.32 and CV² >= 0.49 → erratic

ADI >= 1.32 and CV² < 0.49 → intermittent

ADI >= 1.32 and CV² >= 0.49 → lumpy

5.2 Base forecast methods
Smooth

MA8 = average of last 8 weekly gross quantities

Erratic

For phase 1, keep MA8 but always set review_flag = true unless confidence is high.
Structure code so later we can swap to a blend without UI redesign.

Intermittent / Lumpy

Median6 = median of last 6 weekly gross quantities

New/Sparse

Use simple fallback:

if last 4 weeks contain sales, use average of non-zero weeks in last 4 or 8

else 0
Mark as review

No-demand

0 and review if stock still exists or item is active

5.3 Trend adjustment for smooth items

For smooth items only:

compute MA8

compute AvgLast2

Trend up rule:

if AvgLast2 > MA8 * 1.15

then trend_flag = up

trend_adjusted = min(MA8 * 1.25, MA8 + 0.5 * max(0, AvgLast2 - MA8))

Trend down rule:

if both of the last 2 weeks are below MA8

and AvgLast2 < MA8 * 0.90

then trend_flag = down

trend_adjusted = max(MA8 * 0.75, MA8 - 0.5 * (MA8 - AvgLast2))

Else:

trend_flag = flat

trend_adjusted = MA8

For non-smooth classes, keep trend_adjusted = base_forecast in phase 1.

5.4 Seasonality calculation

Build monthly indices from complete months only.

For phase 1:

calculate at prefix level for all items

calculate at brand level where enough volume exists

only use sku level if at least 2 yearly cycles later; for now keep disabled or very restrictive

Monthly index calculation:

for each level_code and month_no:

month demand / average monthly demand

smooth the raw index toward 1.00

cap usable smoothed index between 0.85 and 1.15

Confidence guidance:

enough months and enough volume → medium

low sample → low

none → none

Seasonality source choice hierarchy:

brand if reliable

prefix if reliable

none

Historical embedded seasonal index:

take the 8 weeks used by MA8

map each week to its month factor

compute weighted average factor across those weeks

Future seasonal index:

use the forward planning horizon days

weight by how many days of the future horizon fall into each month

compute weighted average factor

Then:
final_forecast = trend_adjusted * (future_seasonal_index / hist_embedded_seasonal_index)

If no reliable seasonality source:

both indexes = 1

5.5 Replenishment logic

Use:

target_stock_qty = final_forecast_daily_qty * (cover_days + lead_time_days + review_cycle_days) + safety_stock_qty

net_available_qty = on_hand_qty + incoming_qty - reserved_qty

raw_recommended_order_qty = max(0, target_stock_qty - net_available_qty)

Rounding logic:

respect override order multiple if present

else use case qty where meaningful

enforce MOQ from override or item master

final rounded_order_qty

If item is inactive and still selling:

allow forecast but set review flag

9. Review flags

Set review_flag = true when any of these happen:

demand class is erratic, lumpy, new_sparse

inactive item had sales in last 26 weeks

forecast method changed from last run

trend flag is up or down and impact > threshold

seasonality source changed

seasonality confidence low but adjustment active

forecast change exceeds threshold, e.g. 20%

rounded order differs materially from raw order

missing supplier mapping

missing brand/category where needed

no demand but still stock exists

Persist a readable review_reason.
