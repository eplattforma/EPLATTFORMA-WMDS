# Task #31 — "Send to Batch" During Picking + Batches Section on Dashboard

## What & Why

During regular order picking (`/picker/invoice/<no>/item`), pickers currently have two
options when they cannot pick an item immediately:

- **Skip** — sets `pick_status = 'skipped_pending'`, item returns to the end of the queue
- **Exception** — sets `pick_status = 'exception'`, flags the item for supervisor review

A third option is needed: **"Send to Batch"** — the picker knows someone else will
pick this item later (different zone, needs equipment, wrong shift, etc.) and rather
than skipping it repeatedly or raising an exception, they want it collected into a
dedicated batch that can be picked later by someone else.

All "send to batch" items from the **same route** accumulate in a single shared batch
session automatically — one batch per route, named `DEFERRED-ROUTE-<route_id>`.
This batch lives in the existing batch picking infrastructure and can be claimed and
worked by any authorised picker.

Additionally, the Picking Dashboard needs a **"Batches"** section at the bottom (below
Unassigned) that shows all active deferred batches so managers can monitor and
coordinate follow-up picking.

## Done Looks Like

### During Picking

- The picking screen (`templates/picking.html`) has a **"Send to Batch"** button
  alongside the existing Skip and Exception buttons.
- Tapping "Send to Batch" shows a small confirmation modal with an optional note
  field ("Why are you sending this to batch?"). The note is optional, not required.
- The action:
  1. Sets `InvoiceItem.pick_status = 'sent_to_batch'`
  2. Sets `InvoiceItem.locked_by_batch_id = <deferred_session_id>`
  3. Creates or reuses a `BatchPickingSession` named `DEFERRED-ROUTE-<route_id>`
     for this route (idempotent — one per route)
  4. Adds the item to that batch's queue via `batch_pick_queue` (same flow as
     regular batch creation)
  5. Writes an `ActivityLog` entry: `activity_type = 'send_to_batch'`
  6. Redirects to the next unpicked item on the invoice (same as skip behaviour)
- The item is immediately **hidden from the regular picking list** for that invoice
  (locked by the batch, same as SENSITIVE items).
- On the invoice's item list, "sent to batch" items appear with a purple 🗂️ badge
  showing the batch name: `"In batch: DEFERRED-ROUTE-410"`.
- The invoice's pick progress count (`8/11`) excludes sent-to-batch items from
  the denominator OR counts them as "deferred" — the picker should not be blocked
  from completing the invoice just because some items are in the batch.
- When all non-deferred items are picked, the invoice moves to status
  `awaiting_batch_items` (this status already exists — `routes.py:2966`).
  The invoice remains in this status until the deferred batch is completed for
  those items.

### The Deferred Batch Session

- Named `DEFERRED-ROUTE-<route_id>`, e.g. `DEFERRED-ROUTE-410`
- `session_type = 'deferred_route'` (new type alongside existing `'cooler_route'`
  and `'standard'`)
- `zones` field populated with the zone(s) of the deferred items (comma-separated,
  same as regular batches)
- `picking_mode = 'Sequential'` (standard)
- Status lifecycle: `Created` → `In Progress` → `Completed`
- The batch accumulates items from multiple invoices and multiple pickers on the
  same route — it is **shared and additive**. Any picker on Route 410 can send
  items to `DEFERRED-ROUTE-410` and they all land in the same batch.
- The batch is visible in `/picker/batch/list` and `/picker/batch/<id>` for any
  authorised picker to claim and work.
- Items in the batch are sorted by warehouse location (same as regular batches —
  `wms_aisle`, `wms_bay`, `wms_level`, `wms_position` from `ps_items_dw`).
- When the batch is picked and completed:
  1. `InvoiceItem.is_picked = true` for each item
  2. `InvoiceItem.locked_by_batch_id = null` (unlocked)
  3. `InvoiceItem.pick_status = 'picked'`
  4. Invoice status is recalculated — if all items are now picked, invoice moves
     to `ready_for_dispatch` (or `awaiting_packing` if packing is required).
  5. `ActivityLog` entry: `activity_type = 'deferred_batch_completed'`

### Picking Dashboard — Batches Section

A new section at the bottom of the Picking Dashboard (`/picker/dashboard`), below
the existing Unassigned invoices group. Visible to `warehouse_manager` only (not
individual pickers — they see batches via `/picker/batch/list` already).

Section header: **"Batches"** with a count badge showing active deferred batches.

Each batch row shows:

| Column | What it shows |
|--------|--------------|
| Batch | Batch number (DEFERRED-ROUTE-410) |
| Route | Route name and date |
| Items | Total items / picked items (e.g. 4 / 7) |
| Invoices | Count of invoices with deferred items |
| Status | Created / In Progress / Completed |
| Claimed by | Username of picker who claimed it, or "— unclaimed" |
| Action | View batch / Assign to picker |

Clicking "View batch" opens `/picker/batch/<id>` in the existing batch view.

Clicking "Assign to picker" opens a small modal to assign the batch to a specific
picker (same as the existing batch assignment flow in `batch_picking_manage.html`).

The section only shows `session_type = 'deferred_route'` batches that are not
`Completed` or `Cancelled`. Completed deferred batches are hidden from this section
(they can be found in the batch history).

### S.3 — `get_or_create_deferred_session(route_id)` service function

Add to `services/batch_picking.py` or a new `services/deferred_batch_service.py`:

```python
def get_or_create_deferred_session(route_id: int, creator_username: str) -> BatchPickingSession:
    """
    Returns the active deferred batch session for a route, creating one if needed.
    
    Idempotent: if a non-completed DEFERRED-ROUTE-<id> session already exists,
    returns it. If all existing sessions for this route are Completed or Cancelled,
    creates a new one (DEFERRED-ROUTE-<id>-2, etc.)
    
    The session inherits zones from the items being added to it (populated
    lazily as items are sent to batch, or upfront from the invoice's items).
    """
    existing = BatchPickingSession.query.filter(
        BatchPickingSession.name == f'DEFERRED-ROUTE-{route_id}',
        BatchPickingSession.status.notin_(['Completed', 'Cancelled'])
    ).first()
    if existing:
        return existing
    
    session = BatchPickingSession(
        name=f'DEFERRED-ROUTE-{route_id}',
        batch_number=generate_batch_number(),  # existing helper
        session_type='deferred_route',
        zones='MAIN',  # will be updated as items are added
        picking_mode='Sequential',
        created_by=creator_username,
        status='Created',
    )
    db.session.add(session)
    db.session.flush()  # get the id without committing
    return session
```

### S.4 — `send_item_to_batch(invoice_no, item_code, username, note)` function

```python
def send_item_to_batch(invoice_no: str, item_code: str, 
                       username: str, note: str = '') -> dict:
    """
    Sends one InvoiceItem to the route's deferred batch.
    
    Returns: {session_id, session_name, batch_number, ok}
    """
```

Logic:
1. Load `InvoiceItem` for `(invoice_no, item_code)`
2. Confirm item is not already locked or picked
3. Get `route_id` from `Invoice.route_id`
4. Call `get_or_create_deferred_session(route_id, username)`
5. Set `InvoiceItem.locked_by_batch_id = session.id`
6. Set `InvoiceItem.pick_status = 'sent_to_batch'`
7. Create `batch_pick_queue` row with correct zone, location, qty from `ps_items_dw`
8. Add item to `BatchSessionInvoice` for this invoice if not already present
9. Update session `zones` field to include this item's zone (if not present)
10. Write `ActivityLog` row
11. Commit and return result dict

### S.7 — Invoice completion logic

In `batch_aware_order_status.py` (the function that determines invoice status),
treat `sent_to_batch` items as:

- **Not blocking invoice completion** — the invoice can reach `awaiting_batch_items`
  even with deferred items outstanding
- **Blocking `ready_for_dispatch`** — the invoice cannot be dispatched until the
  deferred batch for those items is completed

Update the `update_order_status_batch_aware` function:

```python
# Items are "complete" if:
# - is_picked = true, OR
# - pick_status = 'sent_to_batch' AND locked_by_batch_id is set
# (deferred items count as "handled" for invoice completion purposes)

# Invoice moves to awaiting_batch_items when:
# - All non-deferred items are picked
# - At least one item has pick_status = 'sent_to_batch'

# Invoice moves to ready_for_dispatch only when:
# - All items are picked (including deferred items picked via batch)
# - No items have pick_status = 'sent_to_batch' remaining
```

## Critical Constraints

- The "Send to Batch" action must set `locked_by_batch_id` so the existing
  `locked_by_batch_id IS NOT NULL → batch_locked` logic hides the item from
  regular picking immediately. Do NOT create a parallel locking mechanism.
- `get_or_create_deferred_session()` must be idempotent — multiple concurrent
  pickers on the same route must not create duplicate deferred sessions. Use a
  database-level upsert or check-then-insert-with-flush pattern with proper
  error handling for race conditions.
- The deferred batch must be visible in `/picker/batch/list` and workable via
  the existing batch picking flow — do not create a separate deferred-only UI.
- `session_type = 'standard'` batches must be completely unaffected — all
  existing batch picking tests must pass unchanged.
- The Batches section on the dashboard must only appear for `warehouse_manager`
  role. Pickers see their own batches via `/picker/batch/list` only.
